# 이커머스 도메인 요구사항 명세서

---

## 1. 개요

- **목적**: 좋아요·주문으로 이어지는 감성 이커머스의 핵심 거래 흐름(브랜드/상품/좋아요/주문)을 정의한다.
- **배경**: 여러 브랜드가 입점한 감성 이커머스 서비스를 가정한다. 방문자는 가입 없이 브랜드와 상품을 둘러볼 수 있고, 로그인한 사용자는 관심 상품에 좋아요를 눌러 두었다가 마음에 든 여러 상품을 한 번에 주문한다. 운영자(어드민)는 브랜드·상품을 등록·수정·삭제하고 전체 주문을 조회해 운영 현황을 파악한다. 좋아요와 주문이 맞물리는 이 거래 흐름이 서비스의 중심이다.

---

## 2. 이해관계자

| 구분 | 설명 | 식별 방식 | 권한 |
|-------|------|-----------|------|
| 비로그인 사용자 | 가입 없이 둘러보는 방문자 | 없음 | 브랜드·상품 **조회만** |
| 로그인 사용자 | 좋아요·주문·쿠폰을 하는 회원 | `X-Loopers-LoginId` / `X-Loopers-LoginPw` 헤더 | 조회 + 좋아요 + 주문 + 쿠폰 발급/사용, **본인 데이터만** |
| 어드민 | 브랜드·상품·쿠폰을 관리하는 운영자 | `X-Loopers-Ldap` 헤더 | 브랜드·상품·쿠폰 관리, **전체 주문 조회** |

## 3. 용어 정의

| 용어 | 정의 |
|------|------|
| 브랜드(Brand) | 상품을 묶는 판매 주체. 하나의 브랜드는 여러 상품을 가진다. |
| 상품(Product) | 판매 단위. 하나의 브랜드에 속한다. 가격과 재고를 가진다. |
| 재고(Stock) | 해당 상품을 지금 몇 개 더 팔 수 있는지를 나타내는 수량. |
| 좋아요(Like) | 사용자가 상품에 표시하는 관심. **한 사용자-상품 쌍당 최대 1개**. |
| 멱등(Idempotent) | 같은 요청을 여러 번 보내도 결과 상태가 동일한 성질. 좋아요 등록/취소에 적용. |
| 주문(Order) | 한 사용자가 한 번에 요청한 구매 단위. 한 개 이상의 주문 항목으로 이뤄진다. |
| 주문 항목(Order Item) | 주문에 담긴 상품 하나와 그 수량. |
| 주문 이력 | 주문 항목이 주문 시점의 상품명·단가를 복사해 보관한 값. 이후 상품 정보가 바뀌어도 이 값은 변하지 않는다. |
| 쿠폰 템플릿(Coupon Template) | 어드민이 정의하는 쿠폰의 원형. 할인 종류(정액/정률)·할인 값·유효일수를 가진다. |
| 쿠폰(내 쿠폰, User Coupon) | 사용자가 템플릿으로 발급받은 쿠폰 한 장. 발급 시점의 혜택·이름·만료일을 복사(스냅샷)해 보관한다. 주문 1건에 1장 사용하며 재사용할 수 없다. |
| 정액 할인(FIXED) | 주문 금액에서 고정 금액(원)을 깎는 방식. |
| 정률 할인(RATE) | 주문 금액의 일정 비율(%)을 깎는 방식. 원 단위 미만은 절사한다. |

## 4. 기능 요구사항

> 각 기능은 "누가·무엇을·왜"를 한 문장으로 적고, 그 아래 **상세 동작 규칙**으로 구체 조건을 정의한다.

### 4.1 브랜드 · 상품 조회 (대고객)

#### US-01: 브랜드 정보 조회
방문자가 특정 브랜드의 정보를 조회해, 그 브랜드가 어떤 곳인지 확인하고 상품을 둘러볼 수 있게 한다.

**상세 동작 규칙**
- **AC-01-1**: 존재하는 브랜드를 조회하면, 브랜드의 고객 노출 정보(이름, 소개 등)를 보여준다.
- **AC-01-2**: 존재하지 않거나 삭제된 브랜드를 조회하면, "브랜드를 찾을 수 없음"이라고 안내한다.

#### US-02: 상품 목록 조회
방문자가 브랜드·정렬·페이지 조건으로 상품 목록을 조회해, 원하는 상품을 효율적으로 찾을 수 있게 한다.

**상세 동작 규칙**
- **AC-02-1**: 아무 조건 없이 목록을 요청하면, 기본 정렬(최신순)과 기본 페이지(첫 페이지, 20개)로 상품 목록을 보여준다.
- **AC-02-2**: 브랜드를 지정해 목록을 요청하면, 해당 브랜드의 상품만 보여준다.
- **AC-02-3**: 정렬 기준(최신순 / 가격 낮은순 / 좋아요 많은순)을 지정하면, 그 기준으로 정렬해 보여준다.
- **AC-02-4**: 조건에 맞는 상품이 하나도 없으면, 빈 목록을 정상적으로 보여준다.

#### US-03: 상품 상세 조회
방문자가 특정 상품의 상세 정보를 조회해, 구매 여부를 판단할 수 있게 한다.

**상세 동작 규칙**
- **AC-03-1**: 존재하는 상품을 조회하면, 상품명·가격·브랜드 정보·좋아요 수·판매 가능 여부를 보여준다.
- **AC-03-2**: 존재하지 않거나 삭제된 상품을 조회하면, "상품을 찾을 수 없음"이라고 안내한다.

---

### 4.2 좋아요 (대고객 · 멱등)

#### US-04: 상품 좋아요 등록
로그인 사용자가 마음에 드는 상품에 좋아요를 눌러, 나중에 다시 찾아보고 자신의 취향이 서비스에 반영되게 한다.

**상세 동작 규칙**
- **AC-04-1**: 아직 좋아요하지 않은 상품에 좋아요를 누르면, 좋아요가 등록되고 그 상품의 좋아요 수가 1 늘어난다.
- **AC-04-2**: **이미 좋아요한 상품**에 같은 요청을 다시 보내면, **오류 없이 성공으로 처리**하며 좋아요 수는 그대로 유지된다(멱등).
- **AC-04-3**: 존재하지 않는 상품에 좋아요를 누르면, "상품을 찾을 수 없음"이라고 안내한다.
- **AC-04-4**: 로그인하지 않은 상태에서 좋아요를 누르면, 요청을 거부한다.

#### US-05: 상품 좋아요 취소
로그인 사용자가 좋아요를 취소해, 더 이상 관심 없는 상품을 정리할 수 있게 한다.

**상세 동작 규칙**
- **AC-05-1**: 좋아요한 상품의 좋아요를 취소하면, 좋아요가 제거되고 그 상품의 좋아요 수가 1 줄어든다.
- **AC-05-2**: **좋아요하지 않은 상품**에 취소 요청을 보내면, **오류 없이 성공으로 처리**하며 좋아요 수는 그대로 유지된다(멱등).
- **AC-05-3**: 좋아요 수는 어떤 경우에도 음수가 되지 않는다.
- **AC-05-4**: **존재하지 않거나 삭제된 상품**에 취소 요청을 보내도, 그 상품에는 좋아요가 없으므로 **오류 없이 성공으로 처리**한다(AC-05-2와 동일하게 멱등). 좋아요 등록(AC-04-3)이 "상품을 찾을 수 없음"으로 거부하는 것과 의도적으로 다르다 — 취소는 "상품이 없으면 좋아요도 없다"는 관계라 좋아요 유무만으로 판정한다.

#### US-06: 내가 좋아요한 상품 목록 조회
로그인 사용자가 자신이 좋아요한 상품 목록을 조회해, 관심 상품을 한곳에서 확인할 수 있게 한다.

**상세 동작 규칙**
- **AC-06-1**: 로그인 사용자가 본인 좋아요 목록을 요청하면, 좋아요한 상품 목록을 페이징하여 보여준다.
- **AC-06-2**: 다른 사용자의 좋아요 목록을 요청하면, 요청을 거부한다(타 유저 정보 직접 접근 불가).

---

### 4.3 주문 (대고객)

#### US-07: 주문 생성
로그인 사용자가 여러 상품을 한 번에 주문해, 원하는 상품을 실제로 구매할 수 있게 한다.

**정상 흐름 요약**: 주문 요청 수신 → 상품·재고 확인 → (쿠폰 지정 시) 쿠폰 검증·할인 계산 → 재고 차감 → 쿠폰 사용 처리 → 주문 생성.

**상세 동작 규칙**
- **AC-07-1**: 주문 요청은 1개 이상의 주문 항목(상품·수량)으로 한다. 존재하지 않는 상품이 포함되거나 수량이 1 미만이면 주문을 거부한다.
- **AC-07-2**: 모든 상품의 재고가 충분하면, 각 상품 재고가 주문 수량만큼 차감되고 주문이 **생성**(`CREATED`) 상태로 저장된다.
- **AC-07-3**: 재고가 하나라도 부족하면, **주문은 생성되지 않으며**, 어떤 상품이 부족한지 안내한다.
- **AC-07-4**: 재고 차감과 주문 생성은 하나의 처리 단위로 묶인다 — 재고 부족으로 주문이 거부되면(AC-07-3) 재고는 변동되지 않는다.
- **AC-07-5**: 주문에는 주문 시점의 **상품명·단가가 이력으로 저장**되어, 이후 상품이 수정·삭제되어도 주문 상세 내역은 변하지 않는다.
- **AC-07-6**: 주문 시 **쿠폰을 0장 또는 1장** 지정할 수 있다(주문 1건당 최대 1장). 쿠폰을 지정하면 주문 시점의 **적용 전 금액·할인액·최종 금액이 이력으로 저장**된다. 쿠폰을 지정하지 않으면 할인액은 0이고 최종 금액 = 적용 전 금액이다.
- **AC-07-7**: 지정한 쿠폰이 **본인 소유가 아니거나, 이미 사용됐거나, 만료됐거나, 최소 주문 금액에 미달하면** 주문을 거부한다. 거부 시 재고·쿠폰 모두 변동되지 않는다(AC-07-4와 동일한 처리 단위).
- **AC-07-8**: 주문이 성공하면 사용한 쿠폰은 **사용 완료(USED)** 상태가 되어 **다시 사용할 수 없다**(재사용 불가). 사용 처리와 주문 생성·재고 차감은 하나의 처리 단위로 묶인다.
- **AC-07-9**: 할인액은 적용 전 금액을 초과하지 않으며, 최종 금액은 0원 미만이 되지 않는다.

> **[가정]** 한 주문에 동일 상품이 중복 포함되면 수량을 합산한다.
> **[가정]** 할인 계산의 원 단위 미만은 절사(floor)한다 — 정률 쿠폰에서 소수점이 나오면 버린다.

#### US-08: 내 주문 목록 조회
로그인 사용자가 기간을 지정해 자신의 주문 목록을 조회해, 과거 구매 내역을 확인할 수 있게 한다.

**상세 동작 규칙**
- **AC-08-1**: 조회 기간을 지정해 목록을 요청하면, 그 기간에 생성된 **본인 주문만** 최신순으로 보여준다.
- **AC-08-2**: 기간을 지정하지 않으면, 전체 기간의 주문을 보여준다.

#### US-09: 단일 주문 상세 조회
로그인 사용자가 특정 주문의 상세를 조회해, 무엇을 얼마에 샀는지 정확히 확인할 수 있게 한다.

**상세 동작 규칙**
- **AC-09-1**: 본인의 주문을 조회하면, 주문 항목별 이력(상품명·단가·수량), **적용 전 금액·할인액·최종 금액**, 주문 상태를 보여준다.
- **AC-09-2**: 다른 사용자의 주문을 조회하면, 요청을 거부한다.

---

### 4.4 브랜드 관리 (어드민)

#### US-10: 브랜드 등록
어드민이 새 브랜드를 등록해, 해당 브랜드의 상품을 판매할 수 있게 한다.

**상세 동작 규칙**
- **AC-10-1**: 필수 항목(브랜드명)을 갖춰 등록하면, 브랜드가 생성된다.
- **AC-10-2**: 필수값이 빠진 채 등록하면, 요청을 거부한다.

#### US-11: 브랜드 정보 수정
어드민이 기존 브랜드 정보를 수정해, 변경된 브랜드 정보를 반영할 수 있게 한다.

**상세 동작 규칙**
- **AC-11-1**: 존재하는 브랜드를 수정하면, 변경 내용이 반영된다.
- **AC-11-2**: 존재하지 않는 브랜드를 수정하면, 요청을 거부한다.

#### US-12: 브랜드 삭제
어드민이 브랜드를 삭제해, 더 이상 취급하지 않는 브랜드를 정리할 수 있게 한다.

**상세 동작 규칙**
- **AC-12-1**: 존재하는 브랜드를 삭제하면, **해당 브랜드의 모든 상품도 함께 삭제된다**.
- **AC-12-2**: 이미 생성된 주문은 이력으로 보존되므로, 브랜드·상품 삭제의 영향을 받지 않는다.
- **AC-12-3**: 브랜드·상품 삭제는 논리 삭제로 처리한다 — 데이터는 보존하되 조회·노출 대상에서 제외한다.

#### US-13: 브랜드 목록 / 상세 조회 (어드민)
어드민이 등록된 브랜드를 목록·상세로 조회해, 브랜드 현황을 관리할 수 있게 한다.

**상세 동작 규칙**
- **AC-13-1**: 어드민이 목록을 요청하면, 등록된 전체 브랜드 목록을 보여준다.

---

### 4.5 상품 관리 (어드민)

#### US-14: 상품 등록
어드민이 특정 브랜드에 상품을 등록해, 고객에게 판매할 수 있게 한다.

**상세 동작 규칙**
- **AC-14-1**: 필수 항목(상품명·가격·초기 재고)을 갖추고 **이미 등록된 브랜드**를 지정해 등록하면, 상품이 생성된다.
- **AC-14-2**: 존재하지 않는 브랜드에 상품을 등록하면, 요청을 거부한다.
- **AC-14-3**: 가격은 0 이상, 초기 재고는 0 이상의 정수여야 한다.

#### US-15: 상품 정보 수정
어드민이 상품 정보를 수정해, 가격·재고 등 변경 사항을 반영할 수 있게 한다.

**상세 동작 규칙**
- **AC-15-1**: 존재하는 상품을 수정하면, 변경 내용이 반영된다.
- **AC-15-2**: **상품의 브랜드는 수정할 수 없다.** 브랜드를 바꾸려는 요청은 거부한다.
- **AC-15-3**: 가격 수정은 **이후 주문부터** 적용되며, 기존 주문 이력에는 영향을 주지 않는다.
- **AC-15-4**: 재고 수정은 재고 수량을 **특정 값으로 설정**하는 것이다(증감이 아니라 절대 수량 반영).

#### US-16: 상품 삭제
어드민이 상품을 삭제해, 판매 종료된 상품을 정리할 수 있게 한다.

**상세 동작 규칙**
- **AC-16-1**: 존재하는 상품을 삭제하면, 고객 조회·주문 대상에서 제외된다.
- **AC-16-2**: 이미 주문에 포함된 상품은 이력 덕분에 주문 상세에 그대로 남는다.

#### US-17: 상품 목록 / 상세 조회 (어드민)
어드민이 등록된 상품을 브랜드별로 조회해, 재고를 관리할 수 있게 한다.

**상세 동작 규칙**
- **AC-17-1**: 어드민이 브랜드를 지정해 목록을 요청하면, 해당 브랜드 상품을 페이징하여 보여준다.

---

### 4.6 주문 조회 (어드민)

#### US-18: 주문 목록 / 상세 조회 (어드민)
어드민이 전체 주문을 목록·상세로 조회해, 운영 현황을 파악하고 문제를 추적할 수 있게 한다.

**상세 동작 규칙**
- **AC-18-1**: 어드민이 주문 목록을 요청하면, **모든 사용자의 주문**을 페이징하여 보여준다.
- **AC-18-2**: 어드민이 특정 주문을 조회하면, 주문자, 항목 이력, **적용 전 금액·할인액·최종 금액**, 주문 상태를 보여준다.

---

### 4.7 쿠폰 (대고객)

> 쿠폰은 **템플릿(원형)** 과 **발급된 내 쿠폰** 두 가지로 나뉜다. 어드민이 템플릿을 만들고(4.8), 사용자가 그 템플릿으로 자기 쿠폰을 발급받아 주문에 사용한다. 발급된 쿠폰은 **발급 시점의 혜택(할인 종류·값)·이름·만료일을 복사(스냅샷)해 보관**하므로, 이후 템플릿이 수정·삭제돼도 이미 발급된 쿠폰의 가치는 변하지 않는다(주문 이력의 상품명·단가 스냅샷과 같은 원칙).

#### US-19: 쿠폰 발급
로그인 사용자가 쿠폰 템플릿으로 자신의 쿠폰을 발급받아, 주문 시 할인에 사용할 수 있게 한다.

**상세 동작 규칙**
- **AC-19-1**: 존재하는(삭제되지 않은) 템플릿을 지정해 발급을 요청하면, 그 템플릿의 **할인 종류·할인 값·쿠폰명**과 **만료일(`발급 시각 + 유효일수`)** 을 복사한 사용 가능(`AVAILABLE`) 상태의 내 쿠폰이 생성된다.
- **AC-19-2**: **한 사용자는 같은 템플릿의 쿠폰을 1장만** 가질 수 있다(1인 1매). 이미 발급받은 템플릿을 다시 발급 요청하면 거부한다.
- **AC-19-3**: 존재하지 않거나 삭제된 템플릿으로 발급을 요청하면, "쿠폰을 찾을 수 없음"이라고 안내한다.
- **AC-19-4**: 로그인하지 않은 상태에서 발급을 요청하면, 요청을 거부한다.

#### US-20: 내 쿠폰 목록 조회
로그인 사용자가 자신이 보유한 쿠폰을 조회해, 무엇을 언제까지 쓸 수 있는지 확인할 수 있게 한다.

**상세 동작 규칙**
- **AC-20-1**: 로그인 사용자가 본인 쿠폰 목록을 요청하면, 보유 쿠폰을 **쿠폰명·할인 내용·만료일·상태**와 함께 보여준다.
- **AC-20-2**: 쿠폰 상태는 **사용 가능(`AVAILABLE`) / 사용 완료(`USED`) / 만료(`EXPIRED`)** 로 구분해 보여준다. 만료는 별도로 저장하지 않고, "사용 가능 상태이지만 만료일이 지난" 쿠폰을 조회 시점에 만료로 판정한다.
- **AC-20-3**: 다른 사용자의 쿠폰 목록을 요청하면, 요청을 거부한다(타 유저 정보 직접 접근 불가).

---

### 4.8 쿠폰 관리 (어드민)

#### US-21: 쿠폰 템플릿 등록
어드민이 쿠폰 템플릿을 등록해, 사용자에게 발급할 쿠폰의 원형을 정의할 수 있게 한다.

**상세 동작 규칙**
- **AC-21-1**: 필수 항목(쿠폰명·할인 종류·할인 값·유효일수)을 갖춰 등록하면, 템플릿이 생성된다.
- **AC-21-2**: 할인 종류는 **정액(`FIXED`, 원 단위) / 정률(`RATE`, 퍼센트)** 두 가지다.
- **AC-21-3**: 정액 할인 값은 1 이상의 정수(원), 정률 할인 값은 1~100 사이의 정수(%)여야 한다. 유효일수는 1 이상의 정수여야 한다.
- **AC-21-4**: 정률 쿠폰에는 별도의 최대 할인 한도를 두지 않는다(비율만 적용).
- **AC-21-5**: **최소 주문 금액(min order amount)** 을 0 이상의 정수(원)로 둘 수 있다(선택, 기본 `0`=제한 없음). 주문 시 적용 전 금액이 이 값 미만이면 쿠폰을 사용할 수 없어 주문이 거부된다(AC-07-7). 이 조건은 발급 시점에 쿠폰으로 복사(스냅샷)되어, 이후 템플릿이 바뀌어도 이미 발급된 쿠폰의 조건은 변하지 않는다.

#### US-22: 쿠폰 템플릿 수정
어드민이 템플릿 정보를 수정해, 이후 발급될 쿠폰에 변경 사항을 반영할 수 있게 한다.

**상세 동작 규칙**
- **AC-22-1**: 존재하는 템플릿을 수정하면, 변경 내용이 반영된다.
- **AC-22-2**: 템플릿 수정은 **이후 발급되는 쿠폰부터** 적용되며, **이미 발급된 쿠폰(내 쿠폰)에는 영향을 주지 않는다**(발급 시점 스냅샷).
- **AC-22-3**: 존재하지 않는 템플릿을 수정하면, 요청을 거부한다.

#### US-23: 쿠폰 템플릿 삭제
어드민이 더 이상 발급하지 않을 템플릿을 삭제할 수 있게 한다.

**상세 동작 규칙**
- **AC-23-1**: 존재하는 템플릿을 삭제하면, 더 이상 발급 대상에서 제외된다(논리 삭제).
- **AC-23-2**: 템플릿 삭제는 **이미 발급된 쿠폰에는 영향을 주지 않는다** — 발급된 쿠폰은 스냅샷으로 자립하므로 만료 전까지 그대로 사용할 수 있다.

#### US-24: 쿠폰 템플릿 목록 / 상세 · 발급 내역 조회 (어드민)
어드민이 등록된 템플릿과 발급 현황을 조회해, 쿠폰 운영 상태를 파악할 수 있게 한다.

**상세 동작 규칙**
- **AC-24-1**: 어드민이 템플릿 목록을 요청하면, 등록된 템플릿을 페이징하여 보여준다.
- **AC-24-2**: 어드민이 특정 템플릿의 발급 내역을 요청하면, 그 템플릿으로 발급된 쿠폰(발급자·상태·만료일)을 페이징하여 보여준다.

## 5. 비기능 요구사항

- **보안/권한**: **본인 데이터 접근 경계**(주문/좋아요/쿠폰 목록)는 명세상 강제한다.
- **정합성**: 주문 생성 시 재고 차감·쿠폰 사용 처리·주문 저장은 하나의 트랜잭션으로 묶여 전부 성공하거나 전부 취소된다(AC-07-4·AC-07-8).
- **동시성(재고)**: 동일 상품에 대한 동시 주문에서도 **초과 판매(oversell)가 발생하지 않는다** — 주문 시 대상 상품 행을 **비관적 쓰기 락**(`@Lock(PESSIMISTIC_WRITE)`, 스냅샷용 조회에 락을 얹음)으로 잠근 뒤 도메인이 `재고 ≥ 수량` 검증 후 차감해 보장한다(AC-07-2·AC-07-3). 동시 주문이 같은 행 락에 직렬화되어 read-modify-write 간극이 사라지므로 lost update가 차단된다.
- **동시성(쿠폰)**: 한 쿠폰이 동시에 두 주문에 **중복 사용**되지 않는다 — `user_coupons`에 **낙관적 락(`@Version`)** 을 두어, `AVAILABLE→USED` 전이가 동시에 일어나면 한쪽만 성공하고 나머지는 충돌로 주문 트랜잭션 전체가 롤백된다(AC-07-8). 한 유저·한 쿠폰의 저경합이라 무는 비용이 거의 없는 낙관적 락을 택했고, 어차피 로드하는 행에 락을 얹는 재고(비관적 락)와 기법을 달리한다.
- **동시성(좋아요 수)**: 동일 상품에 여러 명이 동시에 좋아요/취소해도 **좋아요 수가 정확히 반영된다** — 좋아요 수를 `products.like_count`로 비정규화하고, 행이 실제로 INSERT/DELETE 됐을 때만(영향 행 수 == 1) **원자적 UPDATE**(`like_count ± 1`)로 증감한다. 인기 상품에 몰리는 고경합 단순 카운터라 비관 락(보유 길어짐)·낙관 락(재시도 폭증) 모두 부적합해 원자 UPDATE를 택했다 — 재고(비관 락)·쿠폰(낙관 락)과 또 다른 결이다.
